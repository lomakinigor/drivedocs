# Правила реализации

## Перед написанием кода прочитать
`docs/PRD.md` → `docs/features.md` → `docs/plan.md` → `docs/tech-spec.md` → `docs/data-model.md` → `docs/user-stories.md` → `docs/tasks.md` → `docs/directory-structure.md`

## Для каждой задачи
1. Кратко проанализировать ограничения.
2. Предложить план — не писать код до подтверждения.
3. Перечислить файлы к созданию/изменению.
4. Убедиться, что есть хотя бы один acceptance criterion.
5. Реализовать только запрошенный scope.
6. Объяснить что сделано с указанием F-xxx / T-xxx.
7. Предложить следующий шаг.

## ID-система (обязательно)
Любое изменение, коммит или объяснение, затрагивающее фичу/задачу, обязано содержать ID: `F-xxx`, `US-xxx`, `T-xxx`, `D-xxx`.

## Drivedocs-специфика
- Любое изменение начинается с docs — они единственный источник правды.
- Фронтенд вперёд: mobile-first UX → Zustand stores → Supabase → auth → billing.
- В конце каждого ответа явно перечислить: затронутые ID, обновлённые docs, assumptions.
- Никаких скрытых изменений: всё, что влияет на поведение или архитектуру — в docs с привязкой к ID.
