# Superpowers Workflow — drivedocs

Рабочий цикл для работы с drivedocs совместно с Claude Code.
Каждое значимое изменение проходит следующие фазы.

```
Brainstorm → Spec → Plan → Tasks → Code (+ tests) → Review
```

Мелкие правки (1–3 файла, очевидный scope) допускают сжатый цикл: Plan inline → Code → Review.
Для фич уровня Feature (F-xxx) — полный цикл обязателен.

---

## Phase 1 — Brainstorm

**Цель:** выяснить, нужна ли вообще эта идея и насколько она вписывается в продукт.

**Кто главный:** human

**Что происходит:**
- Человек формулирует идею, проблему или гипотезу.
- Claude помогает найти аналоги в текущем коде, оценить сложность, выявить конфликты с существующими решениями.
- Никаких обязательств — результат может быть "не делаем".

**Docs, которые читаются:**
- `docs/PRD.md` — проверка: вписывается ли идея в MVP-scope
- `docs/features.md` — нет ли похожей фичи

**Gate (что нужно, чтобы идти дальше):**
- Идея сформулирована в одном предложении.
- Нет явного конфликта с PRD или существующими решениями.
- Человек принимает решение: "идём в Spec".

---

## Phase 2 — Spec

**Цель:** зафиксировать, что именно будет сделано и как это выглядит с точки зрения пользователя.

**Кто главный:** shared (human задаёт цель, Claude помогает уточнить acceptance criteria)

**Что происходит:**
- Пишется или уточняется user story в `docs/user-stories.md` (US-xxx).
- Если это новая фича — создаётся запись в `docs/features.md` (F-xxx).
- Acceptance criteria формулируются конкретно (поведение экрана, store-состояние после действия).
- Если нужны архитектурные решения — сразу в `docs/decisions.md` (D-xxx).

**Docs, которые обновляются:**
- `docs/user-stories.md` — новый US-xxx или дополнение к существующему
- `docs/features.md` — новый F-xxx или смена статуса
- `docs/decisions.md` — если принимается новое архитектурное/UX-решение

**Gate:**
- US-xxx существует с acceptance criteria.
- F-xxx зарегистрирован со статусом `planned` или `in-dev`.
- Все открытые вопросы сформулированы явно.

---

## Phase 3 — Plan

**Цель:** определить, как будем реализовывать, какие файлы затрагиваем.

**Кто главный:** AI (при согласовании с human)

**Что происходит:**
- Claude предлагает план реализации: шаги, затронутые файлы, порядок.
- Для фич уровня Feature: `docs/plan.md` обновляется с новой фазой или задачами.
- Для мелких фиксов: план излагается inline в ответе.

**Правило:** код не пишется до согласования плана для задач уровня "фича".

**Docs, которые обновляются:**
- `docs/plan.md` — новая фаза или корректировка Implementation plan

**Gate:**
- Список затронутых файлов согласован.
- Acceptance checks идентифицированы (минимум один).
- Человек дал "go".

---

## Phase 4 — Tasks

**Цель:** разбить план на атомарные T-xxx задачи, пригодные для одного рабочего шага.

**Кто главный:** shared

**Что происходит:**
- Каждая задача записывается в `docs/tasks.md` (T-xxx).
- Задача содержит: тип, описание, статус (todo), Links (F-xxx / US-xxx), Acceptance.
- Задачи нумеруются с запасом (по фазам: T-060, T-061, ...).

**Docs, которые обновляются:**
- `docs/tasks.md` — новые T-xxx со статусом `todo`

**Gate:**
- Каждая задача имеет хотя бы один acceptance check.
- Задачи атомарны (один шаг = один T-xxx).

---

## Phase 5 — Code + Tests

**Цель:** реализовать задачи, выполнить acceptance checks.

**Кто главный:** AI (реализация), human (проверка)

**Что происходит:**
- Claude реализует задачи в порядке зависимостей.
- После каждой T-xxx: статус обновляется на `done` в tasks.md.
- Acceptance checks проверяются вручную или через тесты (см. `docs/testing-strategy.md`).
- Если в процессе возникает архитектурное решение — сначала D-xxx, потом код.

**Правило:** нельзя тихо изменить архитектуру. Любое отклонение от плана — сначала решение (D-xxx или inline note), потом код.

**Docs, которые обновляются:**
- `docs/tasks.md` — статус задач → `done`
- `docs/features.md` — статус фичи → `done` (после последней задачи фичи)
- `docs/decisions.md` — если принято новое решение в процессе реализации

**Gate:**
- Все T-xxx фазы переведены в `done`.
- Acceptance criteria выполнены.
- Нет регрессий в смежных экранах.

---

## Phase 6 — Review

**Цель:** убедиться, что сделанное соответствует spec, и обновить документацию.

**Кто главный:** human

**Что происходит:**
- Человек проверяет фичу по acceptance criteria из US-xxx.
- Claude проверяет, что все docs актуальны (features.md статус, tasks.md, decisions.md).
- При необходимости заводятся follow-up задачи (новые T-xxx в todo).

**Docs, которые проверяются:**
- `docs/features.md` — статус фичи
- `docs/tasks.md` — все задачи фазы done
- `docs/user-stories.md` — acceptance criteria выполнены
- `docs/decisions.md` — новые решения зафиксированы

**Gate:**
- Фича принята человеком.
- Документация синхронизирована.
- Можно делать git commit + push.

---

## Shortcuts (сокращённый цикл)

Для мелких фиксов (1–3 файла, очевидный scope, нет новых решений):

```
Plan inline → Code → Review
```

Примеры: исправить баг в TripCard, добавить пустое состояние, поправить цвет badge.
В этом случае T-xxx всё равно создаётся и закрывается — для traceability.

| Doc | Кто отвечает за актуальность |
|-----|------------------------------|
| `CLAUDE.md` | human (правила) + AI (следует) |
| `docs/PRD.md` | human |
| `docs/features.md` | AI после каждой фичи |
| `docs/user-stories.md` | shared (до реализации) |
| `docs/tasks.md` | AI (обновляет статусы в ходе работы) |
| `docs/decisions.md` | AI (фиксирует при принятии решений) |
| `docs/plan.md` | shared (перед каждой фазой) |
| `docs/tech-spec.md` | human-подтверждение при архитектурных изменениях |
| `docs/data-model.md` | AI (при изменении моделей) |
